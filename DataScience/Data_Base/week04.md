### 1. 데이터 모델

- 데이터 모델:
    - **계층적 데이터 모델 (Hierarchical)**
        - 트리 구조, 1:N
        - 대부분 관계형 데이터 모델로 대체됨
    - **네트워크 데이터 모델 (Network)**
        - N:M
        - 복잡해서 이제 안 씀
    - **개체-관계형 데이터 모델 (Entity-Relationship)**
        - 개체(Entity), 속성(Attribute), 관계(Relationship)을 ERD로 시각적으로 표현
        - 커뮤니케이션 도구로 사용됨
    - **관계형 데이터 모델 (Relational)**
        - Row≥0, Column≥1
        - Row: Record, Tuple
        - Column: Field, Attribute
        - 각 레코드는 기본키(PK)속성에 의해 구별됨
        - 개체-관계형 데이터 모델보다 테이블 간 관계 도출이 더 쉬움
    - **차원 데이터 모델 (Dimensional)**
        - 차원 테이블: 비즈니스 관점 or 분석 대상
        - 팩트 테이블: 비즈니스 측정값
        - 주로 BI의 Data Warehouse, Data Mart에 사용됨
    - **객체지향 데이터 모델 (Object Oriented)**
        - 객체 ⇒ 속성, 동작, 객체
        - 속성과 동작이 동일한 여러 객체를 클래스로 그룹화 가능
        - 새 클래스는 기존 클래스의 속성과 동작을 상속함
        - 복잡한 구조도 모델링 가능, but 안정성 및 성능이 떨어짐 ⇒ 특수 분야에서만 사용됨
    - **그래프 데이터 모델 (Graph)**
        - 노드: 개별 엔티티
        - 엣지: 노드 간의 관계
        - 속성: 노드의 특성
        - 새로운 노드, 엣지 추가 가능
        - 유연하고 쉽게 모델링 가능
        - 네트워크, 계층형 데이터 모델로부터 파생됨
        - 관계 이해 쉬움 ⇒ ‘복잡한 관계’, ‘실시간 쿼리’에 적합

### 2. 데이터 모델링

: 데이터 모델로 변환

<데이터 모델링 프로세스>

1. **요구사항 분석**
    - 시스템 및 사용자의 요구사항을 정의하고 문서화
    - 산출물: 요구사항 분석서, 인터뷰 기록 문서
2. **개념적 데이터 모델링**
    - 특정 DBMS와 독립적인 개념적 스키마 기술
    - 업무 중심적, 포괄적 수준의 모델링
    - 엔티티, 관계 위주의 모델링
    - 산출물:
        1. **주요 속성 도출**: 각 엔티티는 각각의 속성에 대해 단일 속성값을 가져야 함
        2. **식별자 도출**: 각 엔티티를 구분할 수 있는 속성, 엔티티 집합은 반드시 식별자를 가져야 함
        3. **관계 도출**: 존재 관계(소속이나 소유의 관계), 행위 관계(상호작용이나 동작의 관계)
            - **관계의 카디널리티**: 1:1, 1:N, N:1, N:M
            - **관계 이름**: 객체 간 관계가 맺어지는 형태 (관계 시작점, 관계 끝점)
            - **관계 선택 사양**: 선택적 참여 관계, 필수적 참여 관계
3. **논리적 데이터 모델링**
    - 개념적 데이터 모델을 더 상세하게 정의, DBMS에 맞는 논리적 구조로 변환 (In 관계형, 개념적 스키마 → 릴레이션 스키마 변환)
    - 식별 & 비식별 관계
        - 식별 관계(실선): 서로에 대해 필수, 부모 릴레이션의 기본키가 자식 릴레이션의 기본키의 일부
        - 비식별 관계(점선): 서로에 대해 필수 x, 부모 릴레이션의 기본키가 자식 릴레이션의 기본키에 포함 x
4. **물리적 데이터 모델링**
    - 논리적 데이터 모델을 구체적인 데이터베이스 시스템에 맞게 변환
    - 기본키 컬럼 순서, 컬럼명 매핑, 데이터 타입 및 크기, 제약조건, 파티셔닝, 저장구조, 접근 경로 결정
5. **구현 및 유지보수**
    - 물리적 데이터 모델을 기반으로 데이터베이스 생성
    - 운영, 유지보수, 조정, 확장
    - 산출물: 데이터베이스, 데이터베이스 객체 정의문, 데이터 마이그레이션 계획서
6. **검증 및 최적화**
    - 데이터 모델의 정확성, 일관성, 성능 평가 + 수정, 개선
7. **문서화**
    - 각 단계에서 산출된 문서 정리 ⇒ 모델 이해 및 유지보수에 도움

### 3. 정규화 (관계형 데이터베이스)

- 이상: ‘삽입, 삭제, 수정’ 관련하여 발생하는 **불일치 or 오류**
    1. 삽입 이상: 원하지 않는 불필요한 데이터를 반복 입력
    2. 삭제 이상: 삭제할 필요 없는 다른 정보도 함께 삭제
    3. 수정 이상: 일부 레코드값만 수정되어 데이터 일관성이 깨짐
- 함수적 종속성: 릴레이션에서 속성 간의 관계 (X의 값이 주어지면 Y의 값은 항상 유일하게 결정됨)
    1. 완전 함수적 종속성: 기본키에 종속, 기본키가 복합키인 경우 그 속성 집합에 종속
    2. 부분 함수적 종속성: 기본키의 일부에 종속
    3. 이행 함수적 종속성: 3개 이상의 속성 연결, 종속성이 연쇄적
- 정규화: 데이터베이스의 이상 제거 ⇒ 중복 ⬇️, 무결성 ⬆️, 일관성 ⬆️,  유지보수 ⬆️, 논리적 구조 간결, 검색에 효율적
    1. 제1정규형(1NF): 각 속성값은 반복 그룹이 없는 원자 값으로만 구성되어야 함 (i.e. 한 칸엔 하나의 데이터만!)
    2. 제2정규형(2NF): 1NF를 만족하면서 부분 함수 종속성 제거 (i.e. 현재 테이블의 주제와 관련 없는 컬럼은 다른 테이블로 뺌)
    3. 제3정규형(3NF): 2NF를 만족하면서 이행 함수 종속성 제거 (i.e. PK가 아닌 일반 컬럼에만 종속된 컬럼은 다른 테이블로 뺌)
    4. 추가적인 정규화 형식: 제4정규형, 제5정규형, 보이스-코드 정규형 (실무에서는 주로 제3정규형까지만 진행)
- 반정규화: 중복 데이터를 하나 이상의 테이블에 추가하는 데이터베이스 최적화 기술
    - 장점: 조인 개수 ⬇️ ⇒ 검색 쿼리 간단 / 빠른 읽기 작업 수행 ⇒ 검색 성능 ⬆️
    - 단점: 중복 저장 ⇒ 더 많은 스토리지 필요 / 데이터 불일치 발생 / 쓰기 작업에 추가적인 비용 발생
</aside>
